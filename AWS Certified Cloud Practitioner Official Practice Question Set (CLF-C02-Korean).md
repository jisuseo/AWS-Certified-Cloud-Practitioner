# 1. 여러 가용 영역에 Amazon EC2 인스턴스를 사용하여 애플리케이션을 배포하면 어떤 이점이 있는가?


**- 단일 장애 지점을 방지할 수 있다. 가용 영역은 물리적 중복성을 위해 설계 되었으며, 중단 없는 성능으로 복원력을 제공하도록 설계되었다.**

**- 전반적인 가용성이 향상된다**
    
    
- 장애의 영향을 받는 한 곳의 위치에서 모든 인스턴스를 호스팅하면 인스턴스를 전혀 사용하지 못하게 될 수 있다. 가용 영역은 물리적 중복성을 위해 설계되었으며 중단 없는 성능으로 복원력을 제공하도록 설계 되었다. 

- 비용적인 면에서는 증가할 수 있다.
- 여러 리전에서 짧은 지연 시간으로 사용자에게 서비스를 제공하는 가장 좋은 방법은 사용자 위치에 더 가까운 두 번째 리전에서 다른 EC2 인스턴스를 시작하는 것이다. --> 여러 가용 영역과는 관련성이 없다.

# 3. 전 세계 여러 국가에서 고객을 보유한 기업이 AWS 클라우드를 사용하면 어떤 이점이 있는가?
***- 기업은 여러 AWS 리전에 애플리케이션을 배포하여 지연 시간을 줄일 수 있고 글로벌 성능이 향상된다.***
- Amazon CloudFront는 지연 시간을 줄이기 위해 전 세계에 여러 엣지 로케이션을 보유하고 있으며, CloudFront는 데이터, 비디오, 애플리케이션 및 API를 짧은 지연 시간과 높은 전송 속도로 전세계 고객에게 안전하게 제공한는 컨테츠 전송 네트워크 (CDN) 서비스이다.
- AWS Translate는 AWS에서 제공하는 인공신경망 기반 기계 번역 서비스로, 딥러닝 기술을 활용하여 고품질의 언어 번역을 빠르고 경제적으로 제공. AWS 다른 서비스와 통합하여 더욱 강력한 기능을 제공한다. 예를들면, Amazon Comprehend와 함께 사용하여 번역된 텍스트의 감정 분석을 수행하거나, Amazon Polly와 결합하여 번역된 컨텐츠를 음성으로 변환할 수 있다.
서드 파티 웹 사이트 인터페이스를 번역하지는 않는다.
- Amazon Comprehend를 사용하면 여러 언어로 사용자 요청에 응답하는 애플리케이션을 구축하는 서비스가 아닌, 기계학습을 사용하여 텍스트에서 인사이트와 관계를 찾는 자연어 처리 (NLP) 서비스이다. Comprehend는 텍스트를 번역하지 않는다.
- Elastic Load Balancing은 클라이언트에서 들어오는 트래픽을 허용하고, 하나 이상의 가용 영역에서 등록된 대상(예: EC2 인스턴스)으로 요청을 라우팅한다. 여러 리전에 걸친 대상은 허용하지 않는다.
==> 1. 리전 배포의 의미
AWS는 전 세계를 여러 **리전(Region)**으로 나누어 인프라를 제공합니다. 각 리전은 물리적으로 분리된 데이터 센터 집합으로 구성됩니다. 리전 배포는 특정 애플리케이션이나 서비스를 하나의 리전에만 두는 것이 아니라, 여러 리전에 배포하여 글로벌 사용자를 대상으로 하는 경우를 의미합니다.

이점:
지연 시간 감소: 사용자와 가까운 리전에 배치하여 빠른 응답을 제공.
내결함성: 한 리전에 장애가 발생해도 다른 리전에서 애플리케이션을 실행할 수 있음.
==> 2. Elastic Load Balancing(ELB)의 역할
Elastic Load Balancing은 AWS의 서비스로, 트래픽을 여러 서버(인스턴스)로 자동으로 분산합니다. ELB는 단일 리전 내에서 작동하며, 리전에 있는 가용 영역(Availability Zone) 간의 로드 밸런싱도 지원합니다.

이점:
서버 부하를 균등하게 분산하여 장애를 예방.
서버가 증가하거나 감소해도 자동으로 트래픽을 조정.
HTTPS/TLS 지원으로 보안을 강화
==> 3. 리전 배포와 ELB의 관계
ELB는 리전 내부의 트래픽 분산에 초점이 맞춰져 있습니다. 리전 배포와 ELB의 관계를 이해하려면 다음 두 가지 시나리오를 생각해보세요.

(1) 단일 리전 내 배포
하나의 리전 안에 여러 가용 영역을 사용하고, ELB가 트래픽을 리전 내 인스턴스로 분산합니다.
트래픽이 특정 리전으로 집중되어 있어도 ELB가 가용 영역 간의 균형을 맞춥니다.

(2) 다중 리전 배포
여러 리전에 애플리케이션을 배포하는 경우, ELB는 각 리전 내부에서만 트래픽을 분산합니다.
예: 서울 리전의 ELB는 서울 리전의 인스턴스에만 트래픽을 분산.
도쿄 리전의 ELB는 도쿄 리전의 인스턴스에만 트래픽을 분산.
리전 간 트래픽 분산은 Route 53(도메인 네임 시스템) 같은 다른 서비스를 통해 이루어집니다.

# 4. 한 회사에 Oracle 데이터베이스가 실행되는 온프레미스 Linux 기반 서버가 있다. 회사는 AWS Amazon EC2 인스턴스에서 실행되도록 데이터베이스 서버를 마이그레이션하고자 한다. 회사에서 마이그레이션을 완료하려면 어떤 서비스를 사용해야 하는가?
***- AWS Applicaton Migration Service (AWS MGN): 자동화된 리프트 앤 시프트 솔루션이다. 이 솔루션은 물리적 서버와 해당 서버에서 실행되는 모든 데이터베이스 또는 애플리케이션을 AWS의 EC2 인스턴스로 마이그레이션할 수 있다.***
- AWS Database Migration Service (AWS DMS): 온프레미스 데이터베이스에서 AWS의 데이터베이스로 데이터를 마이그레이션 할 수 있다. 실제 서버를 EC2 인스턴스로 마이그레이션하지는 않는다.
- AWS Application Discovery Service는 온프레미스 서버의 사용 및 구성에 대한 정보를 수집하여 AWS로의 마이그레이션을 계획하는데 도움이 된다. (이사 준비를 돕는 서비스) 기존 집(온프레미스 서버)을 분석하여 어떤 가구(애플리케이션)와 물건(데이터)이 있으며, 그 가구들이 서로 어떻게 연결되어 있는지 (종속성)을 파악하고 새로운 집 (AWS)으로 이사 계획을 세우는 데 필요한 정보를 제공



⇒ AWS Application Migration Service (AWS MGN)과 AWS Database Migration Service (AWS DMS)는 AWS에서 제공하는 마이그레이션 도구이지만, 각각 다른 유형의 워크로드와 목적에 초점을 맞추고 있습니다. 차이를 이해하기 위해 핵심 개념과 사용 사례를 정리하면 다음과 같습니다:
1. AWS Application Migration Service (AWS MGN)

         초점: 서버 및 애플리케이션의 전체 환경 마이그레이션 (리프트 앤 시프트).

        작동 방식: 기존의 물리적 서버, 가상 머신(VM), 또는 온프레미스 데이터 센터에서 실행 중인 서버 전체를 AWS의 EC2 인스턴스로 복제 및 마이그레이션.
        
        서버의 운영 체제(OS), 파일 시스템, 애플리케이션, 그리고 데이터를 포함하여 서버 환경 전체를 클라우드로 이동.
        
        결과: 온프레미스에서 실행되던 서버와 동일한 구성을 가진 AWS EC2 인스턴스가 생성됩니다.

        사용 사례: 기존의 서버 및 애플리케이션을 AWS로 그대로 옮겨 클라우드 환경에서 운영하려는 경우.
        
        복잡한 애플리케이션 스택이나 데이터베이스와 결합된 서버를 AWS로 전환할 때.

2. AWS Database Migration Service (AWS DMS)

        초점: 데이터베이스의 데이터만 마이그레이션.

        작동 방식: 온프레미스 데이터베이스, EC2 상의 데이터베이스, 또는 다른 클라우드 서비스에 있는 데이터베이스를 AWS 데이터베이스 서비스(예: Amazon RDS, DynamoDB 등)로 이전.
        
        데이터 전송 중에도 데이터베이스를 계속 사용할 수 있는 실시간 데이터 복제를 지원.
        
        서버나 운영 체제는 이동하지 않음, 단지 데이터만 이동.

        결과: 온프레미스 데이터베이스의 데이터가 AWS 상의 데이터베이스에 저장됩니다.

        사용 사례: 기존 온프레미스 데이터베이스(예: Oracle, SQL Server 등)를 AWS 데이터베이스로 전환하려는 경우.
        
        데이터베이스를 EC2가 아닌 AWS 관리형 서비스로 이동(예: Amazon RDS, Aurora).

핵심 차이

특징

    AWS MGN                                                           AWS DMS
마이그레이션 대상

    전체 서버 환경 (애플리케이션, OS, 데이터 포함)                           데이터베이스의 데이터만
대상 서비스

    AWS EC2 (서버 복제 및 실행)                           AWS 데이터베이스 서비스 (RDS, Aurora 등)
주요 목적

    물리적/가상 서버를 AWS로 그대로 이전                       데이터베이스를 클라우드로 전환 및 통합
사용 사례

    서버 통합 및 재배치                                     데이터 전환 및 클라우드 관리 최적화



⇒ AWS MGN 사용 예:

    회사에서 기존의 온프레미스 애플리케이션 서버를 AWS로 그대로 옮겨 동일한 방식으로 실행하려고 합니다. 서버 전체(운영 체제 + 데이터 + 애플리케이션)를 AWS EC2 인스턴스로 복사 및 실행할 수 있습니다.
AWS DMS 사용 예:

    회사가 기존에 Oracle 데이터베이스를 사용 중인데, 이를 AWS의 관리형 서비스인 Amazon RDS로 이전하고 싶습니다. 데이터베이스 서버는 변경하지 않고 데이터만 옮겨 사용하게 됩니다.
⇒ 비유로 설명:

    AWS MGN: 집(서버 전체)을 다른 도시(AWS EC2)로 통째로 옮기는 것.
        
    AWS DMS: 집을 그대로 두고, 집 안에 있는 가구(데이터)만 다른 집(AWS 데이터베이스)으로 옮기는 것.

# 5. 프리 티어를 초과한 경우 AWS는 AWS Lambda 사용량에 대한 요금을 어떻게 청구하는가?

***⇒ AWS Labmda는 AWS에서 제공하는 서버리스 컴퓨팅 서비스***

    Lambda 요금은 코드를 실행하는데 걸리는 시간에 따라 다르다

    Lambda 함수에 대한 요청 수에 따라 다르다
    
    Lambda 함수의 버전 수는 비용에 영향을 주지 않는다.

    Lambda 함수를 생성하는데 사용되는 프로그래밍 언어는 비용에 영향을 주지 않는다. 

    AWS 계정의 총 Lambda 함수 수는 요금에 영향을 주지 않는다. 

⇒ Lambda는 이벤트가 발생할 때만 켜지고 실행되며, 사용하지 않을 때는 전혀 비용이 발생하지 않는다. 이때는 아무런 리소스를 소모하지 않는다.

# 6. Amazon S3의 특징적인 기능은 다음 중 무엇인가?

1.  Amazon S3은 글로벌 파일 시스템이 아니라, 객체 스토리지 서비스이다.

***2. Amazon S3은 객체 스토어 = 객체 스토리지 서비스 이다.***

3. Amazon S3은 로컬 파일 스토어가 아니라, 객체 스토리지 서비스이다.

4. Amazon S3은 네트워크 파일 시스템이 아니라, 객체 스토리지 서비스이다.

***5. Amazon S3은 내구성이 뛰어난 스토리지 서비스이다. (객체)***

# 7. 한 회사는 AWS 계정 루트 사용자와 관련된 AWS 관리 콘솔 로그인 이벤트를 모니터링하고 이에 대한 알림을 받아야 한다. 회사에서 이러한 요구 사항을 충족하기 위해 사용할 수 있는 AWS 서비스는 무엇인가?

#### ***1. Amazon CloudWatch : 실시간으로 AWS엣 실행되는 애플리케이션 및 AWS 리소스를 모니터링한다. CloudWatch와 AWS CloudTrail을 사용하여 AWS 계정 루트 사용자와 관련된 콘솔 로그인 이벤트를 모니터링하고 이에 대한 알림을 받을 수 있다.***

2. AWS Config : AWS 리소스 구성을 측정, 감사 및 평가할 수 있다. AWS Config는 AWS 계정 루트 사용자와 관련된 콘솔 로그인 이벤트에 대해 알림을 보낼 수 없다.

3. AWS Trusted Advisor: AWS 모범 사례에 따라 리소스를 프로비저닝하는데 도움이 되도록 실시간 지침에 사용할 수 있다. AWS 계정 루트 사용자와 관련된 콘솔 로그인 이벤트에 대해 알림을 보낼 수 없다.

4. AWS Identity and Access Management (IAM) : AWS 서비스 및 리소스에 대한 액세스를 안전하게 관리할 수 있다. IAM은 AWS 계정 루트 사용자와 관련된 콘솔 로그인 이벤트에 대해 알림을 보낼 수 없다

2. Amazon Inspector : Amazon EC2 인스턴스에 배포된 애플리케이션의 보안 및 규정 준수를 개선하는 데 도움이 되는 자동화된 보안 평가 서비스이다. Amazon Inspector는 S3 데이터 분류 및 자동 검색을 수행하지 않는다.

3. Amazon GuardDuty : 악성 활동 및 무단 행위를 지속적으로 모니터링하여 AWS 계정 및 워크로드를 보호하는 위협 탐지 서비스이다. GuardDuty는 S3 데이터 분류를 수행하지 않는다.

4. AWS Secrets Manager : 애플리케이션, 서비스, IT리소스에 액세스하는데 필요한 보안을 지키도록 도와준다. S3 데이터 분류 및 자동 검색을 수행하지 않는다.

# 20. 단일 Amazon S3 버킷에서 정적 웹 사이트를 호스팅하고 있다. 지연시간을 줄이고 전송 속도를 높일 수 있는 AWS 서비스는 무엇인가?

### 1. Amazon CloudFront : CloudFront는 html, css, js 및 이미지 파일과 같은 정적 및 동적 웹 콘텐츠를 사용자에게 더 빨리 배포하는 웹 서비스이다. 콘텐츠는 엣지 로케이션에서 캐시된다. 반복적으로 액세스되는 콘텐츠는 소스 S3 버킷 대신 엣지 로케이션에서 제공될 수 있다.

2. AWS Elastic Beanstalk : Elastic Beanstalk은 용량 관리, 로드 밸런싱, 자동 크기 조정 및 모니터링을 통해 자동으로 배포된 인프라에 공통 프로그래밍 언어로 개발된 웹 애플리케이션 및 서비스를 배포하고 크기 조정하는 서비스이다. Elastic Beanstalk을 사용하면 애플리케이션을 보다 쉽게 프로비저닝하고 지원할 수 있다. 웹사이트 지연 시간을 줄이지는 않는다.

3. Amazon DynamoDB Accelerator (DAX) : 테이블의 응답시간을 한 자릿수 밀리초에서 마이크로초로 줄이는데 사용된다. DynamoDB 테이블은 정적 웹 사이트를 호스팅할 수 없다.

4. Amazon Route 53 : 가용성과 확장성이 뛰어난 DNS 웹 서비스이다. Route 53의 3가지 주요 기능은 도메인 이름을 등록하고, 인터넷 트래픽을 도메인의 리소스로 라우티앟고, 해당 리소스의 상태를 확인하는 것이다. Route 53은 트래픽을 S3 버킷으로 보낼 수 있다. 하지만 질문에서는 S3 버킷을 하나만 설명하므로 Route 53에는 잠재적 경로가 하나뿐이며 지연 시간을 줄일 수 없다.

## 2. 클라우드 컴퓨팅
### 1. Amazon EC2 (Amazon Elastic Computer Cloud)
#### 1. Amazon EC2
 - 안전하고 크기 조정 가능한 컴퓨팅 용량 사용
 - 몇 분 만에 서버 인스턴스 부팅
 - 사용한 만큼만 비용을 지불
#### 2. Amazon EC2 작동 방식
  - 인스턴스 시작 --> 인스턴스에 연결 --> 인스턴스 사용
### Amazon EC2 인스턴스 유형
#### 1. Amazon EC2 인스턴스 유형
- 범용 
  
  - 컴퓨팅, 메모리, 네트워킹 리소스를 균형 있게 제공
  
  - 다양한 워크로드에 적합
- 컴퓨팅 최적화
  - 고성능 프로세서 제공
  - 컴퓨팅 집약적 애플리케이션 및 배치 처리 워크로드에 적합 
- 메모리 최적화
  - 메모리 집약적 워크로드를 위한 빠른 성능 제공
  - 고성능 데이터베이스에 적합
- 스토리지 최적화
  - 낮은 지연 시간 및 높은 IOPS (초당 입출력 작업 수) 제공
  - 분산 파일 시스템 및 데이터 웨어 하우징 애플리케이션과 같은 워크로드에 적합
---
#### 2. 연결하기 활동 : Amazon EC2 인스턴스 유형
- 고성능 데이터베이스에 적합 == 메모리 최적화
- 데이터 웨어하우징 애플리케이션에 적합 == 스토리지 최적화
- 컴퓨팅, 메모리, 네트워킹 리소스를 균형있게 제공 == 범용
- 고성능 프로세서 제공 == 컴퓨팅 최적화
### 2. Amazon EC2 요금
#### 1. Amazon EC2 전용 컴퓨팅
- 전용 인스턴스
  - 단일 고객용 하드웨어의 VPC에서 실행되는 EC2 인스턴스
  - 표준 Amazon EC2 인스턴스에 비해 더 높은 비용
- 전용 호스트
  - 단일 고객용 EC2 인스턴스 용량을 갖춘 물리적 서버
  - 가장 비용이 많이 드는 Amazon EC2 옵션
- 온디맨드
  - 초기 선결제 비용이나 최소 약정 없음
  - 불규칙한 단기 워크로드에 적합
- 스팟
  - 시작 및 종료 시간이 자유로운 워크로드에 적합
  - 온디맨드 대비 비용 절감
- 예약
  - 온디맨드 요금에 비해 결제 할인 제공
  - 1년 또는 3년 약정 필요
- Savings Plans
  - 컴퓨팅 사용량이 일정할 경우 온디맨드에 비해 최대 66% 비용 절감
  - 1년 또는 3년 약정 필요

문제 1. Savings Plan와 스팟 인스턴스의 차이점은 무엇입니까?
- Savings Plans는 1년 또는 3년 기간동안 컴퓨터 사용량이 일정할 경우 사용하면 가격에 이점이 있고, 스팟 인스턴스는 시작과 종료가 자유로운 워크로드에 적합함
---
#### 2. Amazon EC2 Auto Scaling
- Amazon EC2 Auto Scaling
  - 컴퓨팅 요구 사항의 변화에 따라 용량을 조정
  - 동적 조정 및 예측 조정 사용
  - Auto Scaling 그룹에서 최대 4개의 인스턴스를 준비중이며, 최소는 1개, 필요에 따라 확장이 가능하다.
#### 3. Elastic Load Balancing
- Elastic Load Balancing
  - 여러 리소스에 자동으로 트래픽 분산
  - Auto Scaling 그룹에 단일 접점 가능
---
문제 1. 다음은 Auto Scaling과 Elastic Load Balancing 중 무엇의 예입니까?
1. 수요가 적을 때 불필요한 Amazon EC2 인스턴스 제거 == Auto Scaling
2. 온라인 쇼핑몰의 인기 있는 세일 기간 중 두 번째 Amazon EC2 인스턴스 추가 == Auto Scaling 
3. 여러 Amazon EC2 인스턴스에 워크로드 분산 == Elastic Load Balancing
4. 단일 EC2 인스턴스가 전체 워크로드를 처리하지 않아도 되도록 보장 == Elastic Load Balancing
5. 수요에 맞춰 Amazon EC2 인스턴스 수를 자동으로 조정 == Auto Scaling
6. Auto Scaling 그룹으로 들어가는 트래픽의 단일 접점 제공 == Elastic Load Balancing





























### 3. AWS 메시징 서비스
#### 1. 애플리케이션 아키텍처
- 모놀리식 애플리케이션 or 마이크로 서비스
  - 모놀리식 애플리케이션은 각 구성요소가 한 박스안에 들어가 있음
  - 마이크로 서비스는 각 구성요소 박스가 각각 선으로 연결이 되어 있음
#### 2. Amazon Simple Notification Service
-   메시지가 주제에 게시됨
-   구독자는 자신의 주제에 대한 메시지를 즉시 수신
#### 3. 단일 주제에서 업데이트 게시 / 여러 주제에서 업데이트 게시
#### 4. Amazon Simple Queue Service
  - 소프트웨어 구성 요소 간에 메시지를 전송, 저장, 수신
  - 다른 서비스를 사용할 필요 없이 메시지를 대기열에 넣음
#### 5. 예) 대기열에 있는 주문 
-   고객이 주문 --> 주문이 대기열로 들어감 --> 대기열 --> 바리스타가 대기열에서 주문 검색 --> 바리스타가 음료를 만들어 고객에게 제공
#### 6. 예 ) 주문 이행
-   고객이 주문 --> 계산원이 바리스타에게 주문 전달 --> 바리스타가 음료를 만들어 고객에게 제공






### 4. 서버리스 컴퓨팅
#### 1. 서버리스 컴퓨팅
- 가상 서버가 존재하지 않고, 코드만 존재한다.
#### 2. AWS Lambda
- 서버를 프로비저닝하거나 관리하지 않고 코드 실행
- 코드가 실행되는 컴퓨팅 시간에 대해서만 비용 지불
- 다른 AWS 서비스를 사용하여 코드를 자동으로 트리거
#### 3. AWS Lambda 작동방식
- Lambda에 코드  업로드 --> 이벤트 소스에서 트리거되도록 코드 설정 --> 코드는 트리거될 때만 실행됨 --> 사용한 컴퓨팅 시간에 대해서만 비용 지불

## 4. 네트워킹
### 1. 기본 네트워킹 개념 설명
- 고객이 직접 바리스타에게 주문을 할 수 있으면, 빠를 것이다. = 이 개념이 유사하다.
### 2. 퍼블릭 및 프라이빗 네트워킹 리소스의 차이점 서명
#### 1. Amazon Virtual Private Cloud (Amazon VPC)
- Amazon Virtual Private Cloud (Amazon VPC)를 사용하면 고객이 정의한 가상 네트워크 리소스를 실행할 수 있다.
- 퍼블릭 워크스테이션 == 계산원
- 프라이빗 워크스테이션 == 바리스타
#### 2. 서브넷
- 서브넷은 격리된 리소스 그룹을 배치할 수 있는 VPC의 섹션이다. 
- 서브넷은 퍼블릭이거나 프라이빗일 수 있다. 
- AWS 클라우드 > VPC > 퍼블릭 서브넷(Amazon EC2 인스턴스) + 프라이빗 서브넷(데이터베이스)
#### 3. 인터넷 게이트웨이
- 클라이언트가 인터넷을 통해 VPC로 요청 전송
- ==> 게이트웨이를 거쳐서 퍼블릭서브넷으로 진입
#### 4. 가상 프라이빗 게이트웨이
- 사내 데이터 센터에서 인터넷과 VPN을 거쳐서 가상 프라이빗 게이트웨이 를 거쳐 프라이빗 서브넷 (데이터베이스)에 접속
#### 5. AWS Direct Connect
  - 온프레미스 데이터센터와 VPC를 전용 연결설정하는 역할을 한다.
  - AWS Direct Connect 위치 (고객 또는 파트너 라우터-->AWS Direct Connect 엔드 포인트) --> 가상 프라이빗 게이트 웨이 ->퍼블릿 서브넷 / 프라이빗 서브넷

### 네트워크 액세스 제어 목록 및 보안 그룹
#### 1. VPC의 네트워크 트래픽
- 클라이언트 -- 패킷 --인터넷 --> 인터넷 게이트웨이 --> 네트워크 액세스 제어 목록 <--> 퍼블릿 서브넷 (보안그룹)
#### 네트워크 액세스 제어 목록
- 네트워크 액세스 제어 목록 (ACL)은 서브넷의 가상 방화벽이다. 기본적으로 다음과 같이 설정되어 있다.
  - 기본 네트워크 ACL은 모든 인바운드 및 아웃바운드 트래픽을 허용한다.
  - 사용자 지정 네트워크 ACL은 모든 인바운드 및 아웃바운드 트래픽을 거부한다.
#### 상태 비저장 패킷 필터링
- 네트워크 ACL은 상태 비저장 패킷 필터링을 수행한다.
- 패킷이 서브넷에서 나가려면 먼저 아웃바운드 규칙을 기준으로 검사되어야 한다.
#### 보안 그룹
- 보안 그룹은 Amazon EC2 인스턴스의 가상 방화벽이다.
- 기본적으로 보안 그룹은 모든 인바운드 트래픽을 거부하고, 모든 아웃바운드 트래픽을 허용한다.
#### 상태 저장 패킷 필터링
- 보안 그룹은 상태 저장 패킷 필터링을 수행한다.
- 보안 그룹은 들어오는 패킷에 대해 내린 이전 결정을 기억한다.

- 제어 목록 (ACL) == 아파트 단지 입구에 경비원

      - ACL 서브넷 수준에서 트래픽을 제어
      - 경비원은 아파트 단지에 들어오는 모든 사람을 확인하며, 들어오라 또는 돌아가라를 결정
      - ACL은 무조건적인 검사를 수행하며, 각 요청에 대해 허용(Allow) 또는 거부(Deny) 규칙을 평가
      - 트래픽이 들어오거나 나갈 때, 순차적으로 모든 규칙을 확인한 후 최종적으로 결정
      - 특징 : 스테이트리스 (Stateless) : 들어오는 트래픽과 나가는 트래픽을 별도로 관리하며, 연결 상태를 기억하지 않는다. 들어오는 요청과 나가는 응답에 대해 각각 규칙을 만들어야 한다.
      - 주로 서브넷 수준의 트래픽 제어에 사용된다.
- 보안 그룹 (Security Group) == 아파트 개별 현관문

      -  보안 그룹은 개별 인스턴스 수준에서 트래픽을 제어한다.
      -  각 세대의 현관문은 누군가 벨을 누르면, 안에서 누구를 들어오게 할지 결정한다.
      -  보안 그룹은 허용 (Allow) 규칙만 설정할 수 있으며, 모든 트래픽이 기본적으로 차단된 상태에서 허용된 요청만 통과시킨다. 
      -  한번 통과한 요청은 해당 세대와 자유롭게 통신
      -  스테이트폴(Stateful) : 들어오는 트래픽과 나가는 트래픽의 연결 상태를 자동으로 기억한다. 들어오는 요청이 허용되면, 관련된 나가는 응답도 자동으로 허용된다. 
      -  주로 인스턴스 수준의 보안에 사용된다.

- Amazon Route 53은 AWS에서 제공하는 확장 가능하고 신뢰할 수 있는 DNS (도메인 이름 시스템) 웹 서비스이다. Route 53은 인터넷 도메인 이름(: example.com)을 IP 주소(예: 192.0.2.1)와 연결해 사용자가 특정 리소스에 쉽게 접근할 수 있도록 도와준다. 또한 트래픽 라우팅, 상태 확인, 그리고 도메인 등록 서비스도 제공.
- Route 53 : 인터넷 도메인 이름을 IP 주소로 바꿔주는 역할 
- Route 53 : 여러 경로 (서버나 리소스) 중에서 가장 적합한 경로를 선택해 사용자를 안내 - 사용자 위치에 따라 가장 가까운 리전의 서버로 연결하거나 장애가 발생하지 않은 서버로 트래픽을 보내준다.

- Amazon EC2 인스턴스, 데이터가 포함된 인스턴스 스토어
  -  인스턴스 스토어가 연결된 Amazon EC2 인스턴스가 실행중이다.
- Amazon EC2, 데이터가 포함된 인스턴스 스토어
  - ==> 인스턴스가 중지 또는 종료된다.
- Amazon EC2, 데이터가 포함되지 않은 인스턴스 스토어
  - ==> 연결된 인스턴스 스토어에서 모든 데이터가 삭제된다.
#### 3. Amazon EBS 볼륨
- Amazon EC2 인스턴스, 데이터가 포함된 EBS 볼륨
  - ==> EBS 볼륨이 연결된 Amazon EC2 인스턴스가 실행중이다.
- Amazon EC2 인스턴스, 데이터가 포함된 EBS 볼륨
  - ==> 인스턴스가 중지 또는 종료된다. (종료될 경우 EBS 볼륨은 기본적으로 제거된다.)
- Amazon EC2 인스턴스, 데이터가 포함된 EBS 볼륨
  - ==> 연결된 EBS 볼륨의 모든 데이터를 사용할 수 있다.
#### 4. Amazon EBS 스냅샷
- EBS 볼륨 (원본 데이터)
- ==>
  - 1일차 : 모든 데이터가 백업된다.
  - 2일차, 3일차 : 가장 최근 스냅샷 이후에 변경된 데이터만 백업된다.

#### 6. Amazon S3 스토리지 클래스
- S3 Standard
  - 자주 액세스하는 데이터를 위한 설계
  - 최소 3개의 가용 영역에 데이터를 저장
- S3 Standard-IA
  - 자주 액세스하지 않는 데이터에 이상적
  - S3 Standard와 유사하지만 스토리지 가격은 더 낮고 검색 가격은 더 높음
- S3 One Zone-IA
  - 단일 가용 영역에서 데이터를 저장
  - S3 Standard-IA 보다 스토리지 가격이 저렴
- S3 Intelligent-Tiering (지능형 계층화)
  - 액세스 패턴이 불확실하거나 바뀌는 데이터에 이상적
  - 객체당 약간의 월별 모니터링 및 자동화 요금 부과
- S3 Glacier (빙하)
  - 데이터 아카이빙을 위해 설계된 저비용 스토리지
  - 몇 분에서 몇 시간 내에 객체를 검색
- S3 Glacier Deep Archive (빙하 딥 아카이브)
  - 가장 저렴한 객체 스토리지 클래스
  - 12시간 이내에 객체를 검색

### 2. AWS 데이터베이스
#### 1. 데이터베이스 유형
- 관계형 데이터베이스
- 비관계형 데이터베이스
#### 2. 관계형 데이터베이스
- 관계형 데이터베이스에서는 데이터가 다른 데이터 부분과 관련된 방식으로 저장된다.
- 관계형 데이터베이스는 구조화질의 언어(SQL)를 사용하여 데이터를 저장하고 쿼리한다.
#### 3. Amazon Relational Database Service
- AWS 클라우드에서 관계형 데이터베이스를 운영하고 규모를 조정
- 시간이 많이 소요되는 관리 작업을 자동화
- 데이터를 안전하게 저장 및 전송
#### 4. Amazon RDS 데이터베이스 엔진
- Amazon Aurora
  - 엔터프라이즈급 관계형 데이터베이스에 데이터를 저장
  - 불필요한 입출력 (I/O) 작업을 제거하여 데이터베이스 비용을 절감
  - 3개의 가용 영역에 6개의 데이터 복사본을 복제
  (비관계형 데이터베이스)
  - 비관계형 데이터베이스는 행과 열이 아닌 구조를 사용하여 데이터를 구성한다.
  - 예를 들어 키-값 페어에서는 데이터가 항목(키)으로 구성되고 항목은 속성(값)을 갖는다.
- PostgreSQL 
- MySQL
- Maria DB
- Oracle Database
- Microsoft SQL Server
#### 5. Amazon DynamoDB
- Amazon DynamoDB는 서버리스 키-값 데이터베이스이다.
- 용량 변경에 따라 자동으로 확장/축소되어 일관된 성능을 유지할 수 있다.
- 하루에 10조개 이상의 요청을 처리하도록 설계되었다. 
#### 6. AWS Database Migration Service
- 관계형 데이터베이스, 비관계형 데이터베이스 및 기타 유형의 데이터 스토어를 마이그레이션
- 예) MySQL 데이터베이스 (원본) --> AWS Database Migration Service(AWS DMS) --> Amazon Aurora
#### 7. Amazon RDS 및 Amazon DynamoDB
- 각 시나리오에서 사용해야 하는 서비스는 Amazon RDS입니까, 아니면 Amazon DynamoDB입니까
  - Amazon RDS : 관계형 데이터베이스에 데이터를 저장
  - DynamoDB : 키-값 데이터베이스에 데이터를 저장
  - DynamoDB : 서버리스 데이터베이스를 실행
  - Amazon RDS : SQL을 사용하여 데이터를 구성
  - DynamoDB : 하루 최대 10조개 요청으로 확장
  - Amazon RDS : Amazon Aurora 데이터베이스에 데이터를 저장

### 2. AWS Identity and Access Management (IAM)
#### 1. 커피숍에서의 보안
- 계산원이 근무 시작 - 계산원에는 POS 시스템에 액세스할 권한이 있는 계정이 부여 - 계산원은 자신의 계정을 사용하여 POS 시스템에 액세스
#### 2. IAM
- AWS Identity and Access Management (IAM)를 통해 AWS 서비스와 리소스에 대한 액세스를 관리할 수 있다.
    - IAM 기능
      - IAM 사용자
      - IAM 정책
      - IAM 그룹
      - IAM 역할
      - Multi-Factor Authentication
#### 3. AWS 계정 루트 사용자
- AWS 계쩡을 만들면 루트 사용자 자격 증명이 설정 - 첫번째 IAM 사용자를 생성하고 다른 사용자를 생성할 수 있는 권한을 부여 - 새 IAM 사요자로 로그인하고 계속해서 다른 사용자를 생성 / - 제한된 수의 작업에 대해서만 루트 사용자에 액세스
#### 4. IAM 사용자
- IAM 사용자는 AWS 서비스 및 리소스와 상호작용하는 사람 또는 애플리케이션을 나타내는 자격 증명
- 모범사례 : AWS에 액세스해야 하는 각 사용자에 대해 개별 IAM 사용자를 생성
#### 5. IAM 정책
- IAM 정책은 AWS 서비스 및 리소스에 대한 권한을 허용하거나 거부하는 문서
- 모법 사례 : 최소 권한 원칙을 따른다.
#### 6. 예) IAM 정책
- 하단의 샘플 IAM 정책은 ID가 awsdoc-example-bucket인 Amazon S3 버킷의 객체 목록을 보고 액세스할 수 있는 권한을 허용
- {
- "Version" : "2012-10-17", 
- "Statement" : {
- "Effect" : "Allow" , 
- "Action" : [s3:ListObject", "s3: GetObject"], 
- Resource" : ["arn:aws:s3:::awsdoc-example-bucket", "arn:aws:s3:::awsdoc-example-bucket/*"]
- }
- }
#### 7. IAM 그룹
- IAM 그룹은 IAM 사용자의 모음
- 모법사례 : IAM 정책을 개별 IAM 사용자가 아닌 IAM 그룹에 연결
- 구성원은 그룹에 할당된 정책을 상속
#### 8. IAM 역할
- IAM 역할은 임시로 권한에 액세스하기 위해 수임할 수 있는 자격 증명이다.
- 점주는 직원에게 특정 역할로 전환할 수 있는 권한을 부여 - 직원이 '계산원' 역할을 수임. 그러면 직원에게 판매 계산 시스템에 접근할 수 있는 권한이 부여됨 - 직원이 '물품담당자' 역할을 수임. 그러면 직원에게 재고 시스템에 대한 액세스 권한이 부여되고 판매 계산 시스템에 대한 액세스 권한이 취소된다.

#### 9. Multi-Factor Authentication
- Multi-Factor Authentication은 AWS 계정에 추가 보호 계층을 제공
- AWS 웹 사이트에 로그인하려면 사용자가 IAM 사용자 ID와 암호를 입력 - 사용자에게 AWS MFA 디바이스의 인증 응답을 입력하라는 메시지가 표시 - 사용자가 성공적으로 인증되고 요청된 AWS 서비스 또는 리ㅗㅅ스에 액세스할 수 있다.

### 5. 애플리케이션 보안
#### 1. AWS WAF
- 고객의 요청 : 애플리케이션에 액세스하고 싶다 : AWS WAF "차단되지 않은 IP주소에서 왔군요. 들어오시죠"
- 해커의 악성 요청 : 애플리케이션에 댁세스하고 싶다 : AWS WAF "차단된 IP 주소에서 왔군. 들어갈 수 없다."

#### 2. DOS 및 DDOS 공격
- 서비스 거부 공격 : 해커 --> 대상 : 공격이 단일 소스로부터 발생한다
- 분산 서비스 거부 공격 : 해커 --> 봇 --> 대상 : 공격이 여러 소스로부터 발생한다.




#### 3. AWS Shield
- AWS Shield는 분산 서비스 거부 (DDOS) 공격으로부터 보호
- DDoS 공격으로부터 애플리케이션을 보호
- AWS Shield Advanced를 다른 AWS 서비스와 통합
- AWS WAF로 사용자 지정 웹 ACL 규칙을 작성하여 복잡한 DDoS 공격을 완화













#### 4. Amazon Inspector
- Amazon Inspector를 사용하면 애플리케이션에 대해 자동화된 보안 평가를 수행할 수 있다.
- 애플리케이션 보안 평가를 자동으로 수행
- 보안 취약성 및 모법 사례와의 편차를 식별
- 보안 문제 해결 방법에 대한 권장 사항


### 6. 추가 보안 서비스
#### 1. AWS Key Management Service
  - AWS Key Management Service (AWS KMS)는 고객이 사용하면 암호화 키를 사용하여 암호화 작업을 수행하도록 지원.
  - 키에 필요한 특정 수준의 액세스 제어를 선택할 수 있다.



#### 2. Amazon GuardDuty
- Amazon GuardDuty는 AWS 제품 및 서비스에 대한 지능형 위협 탐지 기능을 제공
- Amazon GuardDuty를 활성화 --> GuardDuty가 네트워크 및 계정 호라동을 지속적으로 분석 --> GuardDuty가 지능적으로 위협을 탐지 --> 자세한 결과를 검토하고 조사를 취합
